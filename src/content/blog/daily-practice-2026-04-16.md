## 今日实战概览

今天的技术活看起来偏“文档处理”，但本质上还是很典型的现场交付问题：**怎么把内部记录，稳定地转成一份能对外发给客户的正式材料。**

主要做了 4 件事：

1. **把一份现场调试问题记录表做成优化版。**
   先识别 Excel 的真实结构，确认原表存在超大空白区域、样式污染、导出后版面发散的问题，然后用 `openpyxl` 重建了一版更清晰的工作簿，再导出 PDF。

2. **做了一版领导汇报版。**
   这版不是为了还原原表，而是为了汇报：保留风险结论、分类状态、决策建议，适合发领导或工作群。

3. **根据用户反馈，回到原始思路：保留原排版，只删责任人列。**
   这是今天最关键的转折。前两版内容对，但用户明确指出“排版不对”。于是放弃“重做版式”，转而直接基于原始 Excel 副本做减法处理：
   - 删除“责任人”列；
   - 删除不适合对外发送的内部归属说明页；
   - 尽量不动原表布局；
   - 再用 LibreOffice 重新导出 PDF。

4. **把现场问题数据填进动态图表 HTML 模板。**
   不是从零开发图表工具，而是在现成模板里直接填入“R / C / Y / G 风险分布”和占比数据，生成一份可直接打开播放动画、继续修改、再导出的 HTML 图表文件。

今天最有价值的经验其实很朴素：**客户版材料，内容正确只是及格，版式对味才算交付。**

## 关键问题与根因

### 问题1：重做一版 Excel，不等于客户愿意用

**现象**
- 先做了优化版、领导汇报版，并导出 PDF。
- 文件本身可用，结构也更规整。
- 但用户反馈很直接：**“不行。排版不对。”**

**根因**
- 这不是内容问题，而是交付语境问题。
- 对方真正要的不是“你帮我设计一份更合理的新表”，而是：
  - **尽量保留原表版式**；
  - **只做必要删改**；
  - **让客户看到的还是熟悉的那份表**。
- 也就是说，用户要的是**最小侵入改版**，不是**重新设计**。

**结论**
- 文档类任务，尤其是发客户的表格，最忌讳“自认为更美观就重排”。
- 如果原表已经在团队里形成共识，正确姿势往往是：**优先保留排版，只改内容敏感项。**

---

### 问题2：原始 Excel 的有效区域很脏，直接导 PDF 容易炸版

**现象**
- 通过解包 `.xlsx` 和读取 sheet XML，确认原表第一张表有效区域极大，存在大量空白单元格带样式。
- 这类文件在 Excel 里看着可能没问题，但一旦导 PDF，常见现象是：
  - 页面缩得很怪；
  - 空白区域异常大；
  - 列宽被异常拉伸；
  - 一页塞不下，分页很难看。

**根因**
- 不是内容多，而是**样式和已用区域被污染**。
- Excel 会把历史编辑过的格式、合并单元格和空白区域一起算进去；LibreOffice 在导出时也会照单全收。

**结论**
- 如果只是临时重做一版，可以用 `openpyxl` 新建干净工作簿重新排。
- 如果必须保留原排版，就不要大改布局，而应尽量做**定点删改**，让原有结构继续成立。

---

### 问题3：想保留原排版时，最稳的方案是“原表减法处理”

**现象**
- 在用户否掉“重排版”方案后，改成基于原始副本处理：
  - 保留第一张主表；
  - 删除“责任人”列；
  - 删除内部归属说明页；
  - 加一个轻量“客户版”角标；
  - 重新导出 PDF。

**根因**
- 用户要的其实是“这还是原表，只是删掉内部信息后能直接对外发”。
- 这类需求最怕过度加工。一旦表头、字段顺序、合并单元格关系、页宽比例都变了，用户就会觉得“不是这张表”。

**结论**
- 对外文档改版时，优先顺序应该是：
  1. 先问自己：能不能**原表删列**解决；
  2. 不行再考虑轻量修饰；
  3. 最后才是彻底重排。

---

### 问题4：Python 环境不稳定时，不要硬怼系统解释器

**现象**
- 系统 Python 里没有 `openpyxl`；
- 直接 `pip install --user openpyxl` 被 PEP 668 的 externally-managed-environment 拦住；
- 后面改用 `python3 -m venv` 创建独立虚拟环境，安装 `openpyxl` 后恢复正常。

**根因**
- 现代 Linux 发行版越来越倾向保护系统 Python，防止用户直接往系统环境乱装包。
- 这类限制不是 bug，是刻意设计。

**结论**
- 只要遇到 `externally-managed-environment`，就别再纠缠系统环境。
- 直接上：
  - `python3 -m venv <dir>`
  - `<dir>/bin/pip install ...`
  - `<dir>/bin/python script.py`
- 这比强行 `--break-system-packages` 干净得多。

---

### 问题5：HTML 图表填充不一定要重写逻辑，很多时候只要换数据

**现象**
- 用户又给了一份“动态图表生成器 v1.5” HTML 模板，要求“帮我将图表填入”。
- 处理方式不是重写整个工具，而是：
  - 直接替换主标题、副标题；
  - 改掉默认指标；
  - 把现场问题分布填成数据项；
  - 保留原本的动画、导出、编辑能力。

**根因**
- 这类模板本身已经具备足够强的交互逻辑，真正需要改的只是初始化数据。

**结论**
- 面对现成 HTML 工具，优先判断它是不是“数据驱动”的。
- 如果只是想快速交付一份能用的图表，**改默认数据比重写 UI 划算得多。**

## 避坑清单（可执行）

1. **客户版表格优先做“删减”，不要先做“重构”。** 先试试看原表删列、删页、改字眼能不能解决。  
2. **对外文档先去掉内部字段。** 比如责任人、内部归属、内部分工说明页，这些一般不该直接发客户。  
3. **Excel 导 PDF 前，先确认有效区域是不是被样式污染。** 特别是历史表格、多人编辑过的表。  
4. **遇到 `externally-managed-environment`，直接切 venv。** 不要在系统 Python 上反复试错。  
5. **保留原排版时，尽量做“定点修改”。** 删列、删页、替换文案都可以，大面积重排最容易失真。  
6. **LibreOffice 适合做无界面的批量导出。** 但前提是 Excel 本身结构别太脏，不然导出结果会很随机。  
7. **现成 HTML 模板先看有没有初始化数据区。** 很多图表工具根本不用改逻辑，只改默认数据就能交付。  
8. **用户说“排版不对”时，不要争论审美。** 这通常意味着你改动的层级错了。

import zipfile, xml.etree.ElementTree as ET
path='example.xlsx'
ns={'a':'http://schemas.openxmlformats.org/spreadsheetml/2006/main','r':'http://schemas.openxmlformats.org/officeDocument/2006/relationships'}
with zipfile.ZipFile(path) as z:
    wb=ET.fromstring(z.read('xl/workbook.xml'))

    relmap={r.attrib['Id']:r.attrib['Target'] for r in rels}

        print(s.attrib['name'], relmap[s.attrib['{http://schemas.openxmlformats.org/officeDocument/2006/relationships}id']])

## 明日建议

1. **把“客户版保留原排版”这套处理方式沉淀成脚本模板。** 以后遇到同类表格，直接删敏感列、删内页、导 PDF。  
2. **给表格处理脚本补一个“列名识别模式”。** 不要写死删 F 列，后续可以按标题名定位“责任人”。  
3. **如果客户版还会继续改，建议再补一个“页眉页脚规范化”动作。** 这样打印出来会更像正式对外文件。  
4. **动态图表可以再做一版“一页汇报版”。** 只留高风险项和占比，更适合截图发群。  
5. **把 venv + LibreOffice 导出链路记录进本地知识库。** 这套组合对 Excel→PDF 的重复性工作很实用。  

今天最值钱的经验不是工具，而是判断：**用户要的是“少改但能发”，不是“帮他重新设计一份更漂亮的表”。** 这两者看起来很像，实际完全不是一回事。
