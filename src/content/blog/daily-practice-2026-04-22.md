## 关键问题与根因

### 问题1：为什么今天的重点不是“写文章”，而是“验文章”？

**现象**
- 白天没有新的网络故障、部署事故或排障过程可供展开；
- 但文章发布后，实际做了首页、列表页、文章页三层 HTTP 可达性检查；
- 同时还核对了 Git 最新提交，确认文章不只是本地存在，而是已经被纳入版本历史。

**根因**
- 这类按天触发的任务，并不是每天都会产生值得写长文的新案例；
- 如果只关注“文件有没有生成”“build 有没有成功”，很容易忽略最关键的一步：**页面是否真的已经能访问**。

**结论**
- 当天没有新故障，并不代表没有实战内容；
- 对博客发布链路来说，真正该验收的是“可访问结果”，不是“本地命令跑完了”。

### 问题2：博客自动化任务最容易漏掉哪一步？

**现象**
- 很多自动化流程到 `npm run build` 成功就结束；
- 但 build 成功，只能说明静态产物生成了，不代表页面一定已经被当前服务加载或暴露出来。

**根因**
- 把“构建成功”误当成“发布成功”；
- 缺少一层面向最终结果的验收动作，例如：
  - 首页是否可访问；
  - 列表页是否可访问；
  - 目标文章页是否真的返回 200；
  - Git 最新提交是否包含目标文件。

**结论**
- 发布链路应该至少分成四层：
  1. 文件已生成；
  2. Git 已提交并 push；
  3. build 已完成；
  4. 页面已可访问。
- 少了第 4 层，前面三层都只能算“准备完成”，还不能算“交付完成”。

## 避坑清单（可执行）

1. **不要把 build 成功当成发布成功。**  
   build 只是生成产物，不等于页面已经能访问。

2. **发布后至少检查 3 个 URL。**  
   首页、文章列表页、目标文章页，一个都别省。

3. **文章页校验不要手打路由，最好从文件名推 slug。**  
   `daily-practice-2026-04-22.md` 对应 `daily-practice-2026-04-22`，少出错。

4. **Git 也要做交叉确认。**  
   本地文件在，不代表已经 commit；commit 了，也不代表你检查的是对的文件。

5. **内容少时，不要硬凑长文。**  
   没有新的大块实战，就把当天真实做过的校验动作写清楚。

6. **推送前固定检查远端。**  
   本任务要求远端固定为 `git@github.com:boqin-87/openclaw-node.git`，每次都应核对。

7. **日报类自动化要保留“验收视角”。**  
   不只是记录做了什么，还要记录最后有没有成功对外提供结果。
