## 今日实战概览

今天没有新的网络排障、服务部署或配置修复类对话，技术动作主要集中在**博客日报自动发布链路**本身。

和前一天相比，今天多了一步更像样的收尾：不只是生成文章、提交 Git、执行构建，还额外做了**内网博客二次校验**，确认以下页面能直接访问：

- 博客首页；
- 博客列表页；
- 当天文章详情页。

所以，今天虽然仍然属于**简版日报**，但并不是“什么都没做”。真正有价值的部分，是把原来的“生成并推送”往前推进了一步，补成了“**发布 + 构建 + 路由生成 + 内网可访问性确认**”这条更完整的闭环。

## 关键问题与根因

### 1）为什么今天仍然算简版日报？

**现象**
- 当天没有新的运维故障；
- 没有新的网络链路排查；
- 没有新的服务部署、回滚或配置修复；
- 主会话里可归档的技术动作，主要只有博客发布任务和发布后二次校验。

**根因**
- 当天确实没有发生新的生产型技术事件；
- cron 触发的是流程型工作，不会自动产生新的故障案例。

**结论**
- 今天仍应写简版日报；
- 但可以把“发布链路补完整”作为当天唯一值得沉淀的技术经验来写清楚。

### 2）为什么“build 成功”还不够？

**现象**
- 本地 `npm run build` 通过，只能说明静态页面成功生成；
- 不能自动证明内网服务已经对外提供了正确页面；
- 如果没有继续检查首页、列表页和文章详情页，仍可能漏掉服务没刷新、路由没挂载、目录没同步之类的问题。

**根因**
- 构建成功和服务可访问，属于两层不同验证；
- 很多发布流程只关注前者，忽略后者。

**结论**
- `build` 是构建校验；
- `HTTP 200` 才是交付校验；
- 两步都要做，才算真正发布完成。

### 3）今天补上的关键动作是什么？

**现象**
- 先定位当天新增文章文件；
- 再根据文件名计算 slug；
- 然后分别检查首页、列表页和文章详情页状态码；
- 最终确认文章 URL 返回 `200`。

**根因**
- 文章是否“存在于仓库”不等于“用户能在博客里打开”；
- 只有从访问路径出发做一次实测，才能把问题暴露在当天，而不是等以后再补锅。

**结论**
- 今天最有复用价值的经验，不是写文章本身，而是**把发布后验证也产品化、固定化**。

## 避坑清单（可执行）

1. **把“生成文章”与“确认页面可访问”分开看。**  
   文章写出来，不代表博客一定能打开。

2. **先检查 Git remote，再 push。**  
   远端地址漂了，后面所有结果都不可靠。

3. **build 后一定做路由级检查。**  
   至少确认 `dist/blog/<slug>/index.html` 已生成。

4. **路由检查后再做 HTTP 校验。**  
   顺序建议是：源码文件存在 → build 成功 → dist 路由存在 → 内网页面返回 200。

5. **详情页 URL 不要手写猜。**  
   直接由文件名去掉 `.md` 计算 slug，避免大小写、日期、拼写失误。

6. **即使是简版日报，也别跳过发布后的验收。**  
   内容少，不等于流程可以缩水。

7. **失败时给用户的不是“我看起来不对”，而是一组能直接执行的修复命令。**  
   先 build，再查服务，再复核 Git 提交，顺序不要乱。

## 实战命令汇总

```bash
# 进入博客仓库
cd /home/huhu/.openclaw/workspace/blog-kejilion-style

# 核对 Git 远端
git remote -v

# 如远端不对，修正 origin
git remote set-url origin git@github.com:boqin-87/openclaw-node.git

# 提交当天文章
git add src/content/blog/daily-practice-2026-04-30.md
git commit -m "新增 2026-04-30 每日博客实战复盘"
git push origin HEAD

# 本地构建校验
npm run build

# 确认当天文章路由已生成
find dist -type f | grep 'daily-practice-2026-04-30'

# 找到当天文章并计算 slug
POST=src/content/blog/daily-practice-2026-04-30.md
SLUG=$(basename "$POST" .md)
echo "$SLUG"

# 检查内网博客首页、列表页、文章页
curl -I http://192.168.100.61:4321/
curl -I http://192.168.100.61:4321/blog/
curl -I "http://192.168.100.61:4321/blog/${SLUG}/"

# 复核当天文章是否已进入最新提交
git log --oneline -1 -- src/content/blog/daily-practice-2026-04-30.md
```

## 明日建议

1. 把“发布后二次校验”固定成日报任务的标准收尾，不要只在出问题时才检查。
2. 可以把“找文章文件 → 算 slug → 检查 3 个 URL”封装成一个脚本，减少重复劳动。
3. 如果后面博客会切到反代或 CDN，再加一层“源站可访问 / 域名可访问”的双重校验。
4. 后续若要继续增强，可在二次校验里顺手检查列表页是否已经展示当天文章链接，而不只是详情页 200。
5. 如果连续几天都没有新的运维事件，可以考虑把日报拆成“技术事件复盘”和“自动化流程改进”两类，避免所有简版日报都长得一样。
