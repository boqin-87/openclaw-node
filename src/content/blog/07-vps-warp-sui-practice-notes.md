## 二、今天 ROS 相关的真实问题（按故障类型拆解）

> 这里不是纸面教程，是今天实战里真实踩过的坑。

### 问题 A：WGWARP 试验影响现网链路

现象：

- 在做 `ROS -> VPS -> WARP` 分流实验时，用户反馈：
  - WG-HOME 回家链路受影响
  - 国外访问异常

根因（复盘）：

- 新增路由表 / mangle / NAT / peer 叠加后，触发了与现网策略的交叉影响。
- 问题不在某一条规则，而在“**实验链路和生产链路混跑**”。

处置：

- 做了定向回滚，仅删除 WGWARP 相关新增项：
  - 接口、路由表、mangle、nat、address-list、peer
- 回滚后只读复核，确认现网链路恢复。

规避建议：

1. **实验链路隔离**：只对测试源 IP 生效，绝不直接挂全网。  
2. **命名规范**：所有实验规则统一前缀（如 `WGWARP-*`），回滚可一键定位。  
3. **先回滚脚本再上生产**：所有实验配置必须先准备 rollback 清单。

---

### 问题 B：RouterOS CLI 与 SSH 引号嵌套导致误操作风险

现象：

- 通过 SSH 发 RouterOS 命令时，若 Linux shell 与 ROS CLI 引号嵌套处理不好，容易出现语法错或命令截断。
- 另外，RouterOS CLI 并非 Linux shell，习惯性 `echo` 之类写法会踩坑。

规避建议：

1. 复杂命令优先拆分多条，减少深层引号嵌套。  
2. 先只读校验，再执行写操作。  
3. 写操作统一带 comment，便于 `find comment=...` 回滚。

---

### 问题 C：看似“网络慢”，实际是链路占用/策略冲突

现象：

- 链路测速异常并不总是隧道坏了。
- 实战中曾出现上传任务挤占上行，导致测速结果误判。

规避建议：

- 把排障顺序固定为：
  1. 接口握手状态
  2. 路由命中
  3. 防火墙计数器
  4. 实际带宽占用
- 先排“资源占用”再排“协议错误”，能省大量时间。

## 三、ROS 配置前中后：一套稳态操作法

## 0) 变更前（必须做）

- 导出现有配置快照（不含敏感导出分享）
- 列出现网关键对象：
  - WireGuard 接口/peer
  - routing rule / route table
  - mangle / NAT / filter
- 标注“本次变更作用域”：只改测试设备，不碰全局默认路由

## 1) 变更中（灰度）

- 仅给测试源地址打标，按标记路由到实验出口
- 不改现网默认出口
- 每改一步就做一次可观测验证：
  - 能否握手
  - 路由命中是否符合预期
  - 回家链路是否仍正常

## 2) 变更后（收口）

- 若通过：把临时规则转正式命名
- 若失败：按 comment 前缀定向回滚，不做“全删重建”
- 记录结论：是规则冲突、链路质量、还是资源占用

## 四、ROS + Cloudflare 配置指南（脱敏版）

以下给的是可直接套用的结构化指南，避免泄露任何密钥。

### A. DNS 记录分层规划

把域名按用途分 3 层：

1. **面板/业务入口（A/AAAA）**
   - 需要真实回源时用 DNS only
   - 仅做 Web 防护时可考虑 Proxied（视协议而定）

2. **隧道类入口（CNAME 指向 tunnel）**
   - 典型是 `xxx -> <tunnel-id>.cfargotunnel.com`
   - 由 cloudflared 承接，不走传统端口暴露

3. **对象存储/第三方服务映射（CNAME）**
   - 如 R2 公共域名映射，注意缓存与回源策略

关键原则：

- 代理状态（Proxied / DNS only）必须与业务协议匹配。
- 不要在文档里放 API Token；只写“凭证文件路径规范”。

---

### B. ROS DDNS 与 Cloudflare 联动（模板思路）

建议机制：

- ROS 侧定时任务获取当前公网 IP
- 仅当 IP 变化时才调用 Cloudflare API 更新对应 A 记录
- 失败要有日志与重试退避

必做保护：

1. Token 权限最小化（仅单 Zone DNS 编辑）  
2. Token 不落明文到公共仓库  
3. 更新脚本打印“结果码 + 记录名 + 新IP”，不打印鉴权头

---

### C. 端口与反代协同（重点坑位）

- Web 面板通常占用 80/443
- TUIC 等 UDP 协议必须提前避开 443/UDP 冲突
- 如果启用 Caddy HTTP/3，请默认认为 443/UDP 已被占用

建议：

- Reality 用独立 TCP 端口
- TUIC 用独立 UDP 端口
- 文档里固定写“端口职责表”，避免后期互相覆盖

---

### D. 证书失败时的排查顺序

1. 本地反代是否可达（127.0.0.1 验证）  
2. 域名解析是否到正确回源  
3. 80/443 的实际入口是否命中预期主机  
4. 失败再切 DNS-01（而不是无脑重试 HTTP-01）

今天最有价值的不是“配成了什么”，而是把“为什么会翻车、翻车后怎么稳住”沉淀成了可复用流程。
