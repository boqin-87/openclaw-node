## 今日实战概览

今天真正有技术含量的内容，基本都集中在一件事上：**查 RouterOS 日志，看外网有没有可疑 IP。**

表面上只是用户一句“把 log 发我，看看有没有可疑 IP”，但拆开看，其实包含了几步很典型的运维动作：

1. **先只读，不改配置。** 这符合当前设备排障约定：先检查，再决定要不要动防火墙或服务。
2. **从本地安全读取 RouterOS 凭据并发起 SSH。** 只拿到必要访问权限，不把敏感信息写进输出。
3. **拉三类数据一起看：**
   - 系统日志 `/log`
   - 当前连接表 `/ip firewall connection`
   - 设备基础状态 `/system resource print`
4. **把“真正异常”和“正常业务流量”分开。** 比如 Winbox/L2TP 探测、SSH 公网碰撞和 51413 端口入站，性质完全不同，不能混着看。
5. **先给出候选可疑 IP，不直接封。** 因为公网来源里可能混有你自己的出口 IP 或正常业务对端，先判断再下手才稳。

今天这次排查的价值，不在于发现了多复杂的攻击，而在于把一个很常见的家庭/自建网络现实说清楚了：**只要公网端口开着，被扫是日常；重点不是“有没有扫”，而是“哪些真要管、哪些只是背景噪音”。**

## 关键问题与根因

### 问题1：日志里到底有没有可疑公网 IP？

**现象**
- 用户明确要求查看 ROS 日志，并判断是否存在可疑 IP。
- 读取到的系统日志里，出现了两类非常典型的外网探测记录：

```text
denied winbox/dude connect from 146.88.240.23
denied winbox/dude connect from 81.29.142.100
```

以及两条 L2TP 首包探测：

```text
first L2TP UDP packet received from 146.88.241.82
first L2TP UDP packet received from 198.235.24.84
```

**根因**
- 这不是内网异常，也不是系统误报，而是**公网暴露服务后被外部扫描/试探**的正常现象。
- 尤其是 Winbox、L2TP、SSH 这类端口，只要对公网可达，就很容易收到自动化探测。

**结论**
- 今天可以明确标记的可疑公网 IP 至少包括：
  - `146.88.240.23`（Winbox/Dude 探测）
  - `81.29.142.100`（Winbox/Dude 探测）
  - `146.88.241.82`（L2TP 探测）
  - `198.235.24.84`（L2TP 探测）

这里最重要的一点不是“这些 IP 很坏”，而是：**它们的行为已经越过了“普通业务访问”的边界，进入了明确的服务探测。**

---

### 问题2：SSH 22 端口确实在被公网碰

**现象**
连接表里可以看到多个公网地址访问本机公网 22 端口，例如：

```text
61.164.113.57:* -> 公网IP:22
183.36.249.8:* -> 公网IP:22
```

大多表现为 `close` 状态，没有形成成功会话。

**根因**
- 这说明 SSH 服务对公网可见，或者至少 NAT/策略层面对外暴露了 22。
- 一旦暴露，就会不断收到互联网背景扫描、撞库和弱口令尝试。

**结论**
- 当前没有看到成功入侵迹象，但**“有人在碰 22”这件事是确定的**。
- 这类记录的意义不在于单条连接危险，而在于它说明你的暴露面是真实存在的。

---

### 问题3：51413 端口流量很多，不一定是攻击，但必须知道它在干什么

**现象**
连接表里出现了大量：

```text
随机公网IP -> 公网IP:51413
```

其中还有个别连接处于 `established`。

**根因**
- `51413` 很像 PT / BT / 某类 P2P 服务的常见端口。
- 如果这是你主动开放的业务端口，那么这些入站很可能是业务流量。
- 如果这不是你有意暴露的端口，那就说明：
  - 要么存在 NAT 暴露；
  - 要么某台内网服务在对公网提供 P2P 类服务；
  - 要么配置遗留，自己已经忘了。

**结论**
- `51413` 不能直接判为攻击，但它一定是今天最该继续追的端口之一。
- 真正要做的不是先封，而是先确认：**这个端口是你故意开的，还是忘了它为什么开着。**

---

### 问题4：很多日志其实不是安全事件，而是“噪音”

**现象**
系统日志里大量重复出现：

```text
cloud-ddns-watch: updated 域名 -> 公网IP
user boqin logged in from 内网地址 via ssh
user boqin logged out from 内网地址 via ssh
```

**根因**
- 这类日志本身不是异常：
  - DDNS 更新脚本属于定时任务行为；
  - 内网地址登录是正常管理动作；
  - DHCP 分配/回收地址也是常规噪音。
- 如果直接把全量日志原样发给用户，真正有用的告警会被淹没。

**结论**
- 安全排查时，日志不是越多越好，而是要先去噪、再聚焦。
- 真正需要优先看的，是：
  - `denied`
  - `login failed`
  - `first L2TP UDP packet`
  - 非预期公网源访问管理端口

## 避坑清单（可执行）

1. **先看行为，再看 IP。** 同一个 IP 访问业务端口和扫描管理端口，性质完全不同。  
2. **不要看到公网来源就立刻封。** 先确认是不是你自己的出口、VPN 节点、P2P 对端或正常业务流量。  
3. **Winbox / SSH / L2TP 这类管理入口，如果不需要公网直连，就别裸露。** 比事后拉黑更省心。  
4. **连接表要和系统日志一起看。** 单看 `/log` 容易漏掉当前活跃连接，单看 connection table 又不知道上下文。  
5. **遇到 51413 这类高噪声端口，先确认业务归属。** 是主动开放的 PT/P2P 端口，还是遗留暴露，处理方式完全不同。  
6. **只读排查阶段不要顺手改防火墙。** 先输出证据、列候选 IP、确认暴露面，再决定是否封禁。  
7. **把“登录成功”和“端口被碰”分开看。** 有扫描不等于已经失守，但有暴露就值得收口。

## 实战命令汇总

```bash
# 1) 读取 RouterOS 全量日志（只读）
ssh user@router '/log print without-paging detail'

# 2) 查看连接表，确认哪些公网 IP 正在打到你的公网口
ssh user@router '/ip firewall connection print without-paging'

# 3) 查看路由器基础资源状态
ssh user@router '/system resource print; /system health print'

# 4) 快速过滤管理面相关日志（适合手工筛）
/log print where message~"login|failed|denied|L2TP|winbox|ssh"

# 5) 查看 RouterOS 暴露了哪些服务
/ip service print

# 6) 查看防火墙过滤规则与 NAT 规则
/ip firewall filter print without-paging
/ip firewall nat print without-paging

# 7) 如果要进一步确认 SSH 是否对公网开放
# 从外部主机侧检查 22/8291/1701/500/4500 等端口是否可达
nmap -Pn <your-public-ip> -p 22,8291,1701,500,4500,51413
```

## 明日建议

1. **继续做一轮只读核查：** 把 `/ip service print`、`/ip firewall filter print`、`/ip firewall nat print` 拉出来，确认公网到底暴露了哪些管理口。  
2. **重点确认 51413 的业务归属。** 如果是 PT/P2P 服务，保留但要知道是谁在用；如果不是，就该收口。  
3. **如果 SSH 不需要公网直连，优先改成仅内网 / 仅跳板 / 仅 VPN 访问。** 这比事后封一堆扫描 IP 更有效。  
4. **考虑为 Winbox、SSH、L2TP 加地址列表与速率限制。** 不一定要大改架构，但最基础的入站限制值得补上。  
5. **后续如果用户确认了哪些 IP 不是自己人，可以再做黑名单封禁。** 今天先判断，明天再决定要不要动规则，节奏是对的。  

今天这次排查很像很多家庭网络和自建环境的缩影：**真正的问题不是互联网上有人扫你，而是你自己是否清楚哪些门开着、为什么开着、谁该能进。**
