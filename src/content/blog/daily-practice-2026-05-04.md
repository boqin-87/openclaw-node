## 今日实战概览

今天不是空白日报，实际有 3 组比较典型、而且都能复用的运维实战。

### 1）Vaultwarden 域名访问修复
用户反馈 `bw.6661993.xyz` 不通。最后确认不是 Vaultwarden 容器挂了，也不是 nginx 没监听，而是 **Cloudflare DNS 没有及时跟随家庭公网 IP 变化更新**。同时，内网从公网域名回家访问这条路径，还需要 **Hairpin NAT** 才能真正打通。

最终做了两件事：
- 修 RouterOS 上的 Cloudflare DDNS 脚本，让它不只更新主域名，也同步更新 `bw.6661993.xyz`；
- 补上针对 `8445 -> 内网 443` 的回环 NAT 规则，以及内网访问家庭公网 IP 的通用 Hairpin NAT 规则。

修完后，内网访问 Vaultwarden 外网入口已经恢复正常。

### 2）Surge 回家访问 NAS 低速排查
用户反馈：公网 IP 直接访问 NAS，速度正常；但在 Surge 里命中 `WG-HOME` 或 `SS-HOME` 回家访问内网时，吞吐只有大约 `5 Mbps`。

当天没有直接把问题彻底修掉，但已经排除了一个很常见的误判：**这不是 NAS 本体性能问题，也不像是简单的端口映射或家庭上传带宽问题。**

另外还试过把 `WG-HOME` 的 MTU 从 `1420` 调低，结果速度更差，说明问题并不是一句“把 MTU 调小”就能解决。

### 3）Unraid 安全关机
用户要求关闭 Unraid。这里没有直接一把梭关机，而是先查运行中的虚拟机，再按顺序优雅停机。

实际处理顺序是：
- 先只读检查当前状态；
- 发现还有 3 台虚拟机在运行；
- 逐台执行 `virsh shutdown`；
- 等虚拟机都停掉以后，再对宿主机下发关机命令。

这比直接强关靠谱得多，尤其适合挂着虚拟机和 Docker 服务的家庭 NAS/宿主机环境。

## 关键问题与根因

### 1）Vaultwarden 域名不通，根因不在应用，而在 DNS 更新链路不完整

**现象**
- `bw.6661993.xyz` 不可访问；
- Vaultwarden 容器正常；
- 本机 `127.0.0.1:8072` 正常；
- 本机 `443` 上的 nginx 也能正常响应域名请求。

**根因**
- Cloudflare DNS 没有及时更新到最新家庭公网 IP；
- RouterOS 上现有 DDNS 脚本只覆盖 `6661993.xyz`，没有覆盖 `bw.6661993.xyz` 这个实际业务域名。

**结论**
- 应用层没挂，不代表外部访问链路没问题；
- 做家庭服务外网暴露时，DDNS 脚本必须覆盖所有实际使用的主机名，不能只更新根域名。

### 2）外网域名在内网不通，根因是缺 Hairpin NAT

**现象**
- 即使域名解析恢复后，内网用外网域名访问服务，仍然可能失败；
- 典型场景是：从 `192.168.100.0/24` 里访问家庭公网 IP，再转回内网服务。

**根因**
- 路由器没有为“内网设备访问本家庭公网映射服务”准备回环 NAT；
- 这类流量不经过额外的 srcnat / dstnat 处理时，常见结果就是连接异常或回包路径不对。

**结论**
- 家庭服务一旦既要外网可访问，又要内网也能通过同一个域名访问，Hairpin NAT 基本就是标配。

### 3）Surge 回家慢，阶段判断指向隧道链路，而不是 NAS 本体

**现象**
- 直连公网 IP 访问 NAS 时，速度正常；
- 通过 `WG-HOME` 或 `SS-HOME` 回内网时，速度只有约 `5 Mbps`；
- 下调 `WG-HOME` MTU 后反而更慢。

**根因（阶段性）**
- 现有证据更像是“Surge -> 回家隧道 -> 内网访问”这段路径的问题；
- 不像是 NAS、端口映射、家庭上传带宽这些底层资源本身不足；
- 也不像单纯 MTU 设置过大的单点问题。

**结论**
- 下一步要重点盯 MSS、分流路径、不同回家协议的真实吞吐，以及播放器与大文件直链下载之间的差异。

### 4）Unraid 关机不能只看命令回显，要看实际离线结果

**现象**
- 宿主机关机命令返回了 `powerdown scheduled`；
- 但约 8 分钟后，主机仍能 ping 通，SSH 仍可连接。

**根因**
- “关机已计划”不等于“设备已下线”；
- 虚拟机、文件系统 flush、Docker 收尾或者平台自身的关机流程，都可能拖延实际离线时间。

**结论**
- 关机操作必须分成两段看：
  1. 命令是否成功下发；
  2. 设备是否真的离线。

## 避坑清单（可执行）

1. **域名不通先分层排查。** 先看 DNS，再看反代，再看容器，不要一上来就重启应用。
2. **DDNS 脚本不要只更新根域名。** 业务真正用到的二级域名，也要明确覆盖。
3. **家庭服务既要外网可用又要内网同域名可用时，提前配 Hairpin NAT。** 不要等用户反馈“外面能开、家里反而打不开”再补。
4. **回家链路慢时，先做对照实验。** 直连公网 IP、仅走 `WG-HOME`、仅走 `SS-HOME`、播放器访问、大文件直链下载，分开测。
5. **不要迷信 MTU 是万能解。** 调小后更差，往往说明问题不在这里，别在单点参数上死磕。
6. **关 Unraid 前先清点虚拟机。** 宿主机不是普通单机，直接关机会把虚拟机状态打断。
7. **看到 `powerdown scheduled` 也别当任务结束。** 一定要复查 ping、SSH 或管理面板状态，确认主机真的下线。
8. **写日报时去掉真实凭据、token、密码和带鉴权参数的地址。** 经验可以写，敏感信息不要落文档。

## 明日建议

1. **Vaultwarden 这块建议继续补自动化。** 不只是修当前脚本，而是把“所有业务域名列表”集中维护，避免以后新增域名又漏更新。
2. **Surge 回家低速建议做标准化测速矩阵。** 把网络类型、协议、文件大小、测速方式统一起来，不然很难判断瓶颈到底在哪一段。
3. **Unraid 关机建议补一个自动脚本。** 先列出运行中虚拟机、逐台优雅停机、循环等待、超时报警、最后再关宿主机，这样以后不用每次手动盯。
4. **今天这 3 类问题其实都在提醒同一件事：** 家庭运维里最容易出问题的，不是单个服务本身，而是“服务之间的连接边界”——DNS、NAT、隧道、反代、虚拟化收尾，真正的坑大多在这些地方。
